# Matt Pocock’s Agentic Engineering Workflow (just copy him)

> https://www.youtube.com/watch?v=nQwJVHCtDDY

## 一句話總結
Matt Pocock 認為在 AI 時代真正的競爭力不在於追逐最新模型，而在於打磨「harness」（提示、技能、程式碼庫與工作流程），用扎實的軟體基本功與策略性思維，把 AI 當成可無限調度的戰術程式設計師大軍。

## 重點摘要
- **harness 比模型更重要**：大家都迷戀模型（F1 賽車的引擎），但 harness（底盤、空氣力學）同樣關鍵，而且你對 harness 的掌控度遠高於模型；給對提示、對的技能、改善程式碼庫，才是發力點。
- **AI 吃掉了「戰術程式設計」**：寫程式、抓語法、修 bug 這類日常工作 AI 已經做得比人便宜又快；人類該專精的是「策略程式設計」——架構設計、任務拆解、模組介面、測試與恰到好處的文件。
- **你的技能是 AI 能力的天花板**：AI 讓資深開發者強 10 倍，對初階只有小幅加成；想用好 AI 就要精進自己的領域知識，而不是把思考外包給模型。
- **teach 技能（stateful skill）**：依使用者的「mission」量身打造學習路徑，產出本地儲存的 HTML 課程、cheat sheet、測驗與學習紀錄，運用「近側發展區」與「知識／技能／智慧」等教學原理，實現個人化教育。
- **好技能的兩種型態**：procedure（使用者主動呼叫，如 grill me、two PRD）vs ability（模型自行調用，如編碼規範）；Matt 偏好 procedure，自己掌握方向盤，不把思考交給模型，也避免上百個技能描述塞爆 context window。
- **knowledge / skills / wisdom 三層**：知識與技能可打包成可重用技能分享給團隊，但「智慧」（知道何時做、如何融入真實情境）幾乎必須親身在實際脈絡中累積。
- **AFK 與「佇列」而非「迴圈」**：Matt 用 Sand Castle 在沙箱中跑 AFK（離開鍵盤）代理並結合 GitHub Actions 平行化工作；他認為流行的「agentic loop」被誇大，開發本質是一個「任務佇列」，由人類像國王般排序優先、把關。
- **打造自我改善的系統**：與其驚嘆強模型抓到深層 bug，不如反問「為什麼會有這個 bug」，建立每日 cron 安全掃描、測試、審查等迴圈；便宜模型搭配好 harness 也能找到同樣問題。
- **降低 token 花費的真正方法**：讓程式碼庫更容易修改、架構更好、護欄更完善，這樣笨一點、便宜一點的模型就能完成同樣工作。
- **AI 不擅長的事**：原創構想與產品方向。建商業產品的基本功沒變——去跟真實顧客對話、找出他們要什麼；人類必須掌握願景，決定加哪些功能（甚至問 AI 該「移除」什麼以求簡化）。

## 關鍵結論／takeaways
- **專注 harness 與基本功，別追新模型**：堅守 30、40 年來證實有效的軟體工程原則，保持工具與模型無關（agent-agnostic），新模型出來也能繼續受用；別圍繞單一模型過度最佳化。
- **新模型發布先別急著換**：等約一個月觀察 latency、成本、可用性再決定，不要被一鍵 one-shot 的炒作淹沒。
- **培養策略性思維、停止當 code monkey**：把戰術工作交給 AI，自己專注架構、路線圖與產品決策，把更多思考拉回自己手中。
- **善用 AFK 平行化**：學會把實作委派給沙箱中的離開鍵盤代理，初期設定費點工，之後產能會暴增。
- **用「佇列＋人類把關」模式管理代理**：像國王治國一樣排序任務、設置 human-in-the-loop 檢查點，並逐步把檢查點往產出端推，但別完全移除——你同時也在審查「產出程式碼的系統」。
- **練習語音聽寫（dictation）**：對開發者是被嚴重低估的加速技能（Matt 用 Whisper Flow）。
- **今天就能做的一步**：刪光所有 skill、plugin、MCP server、CLAUDE.md、agents.md，回到空白狀態觀察代理的原始行為，再只疊加你親自挑選的 procedure 技能，並以可自訂、可實驗的方式安裝。
- **用 grill me 式對話取代盲目 one-shot**：描述願景後，請 AI 列出最關鍵的設計／架構／產品決策並反向訪問你，直到雙方達成共識再動工。
